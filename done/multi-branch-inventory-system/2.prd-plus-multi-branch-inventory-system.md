# PRD Review - Multi-Branch Inventory & Order Management System

## 1. Critical questions

### Core objectives

- Who is the primary buyer persona driving requirements (operations lead, branch manager, or finance)?
- Which three workflows must be best-in-class on day one?
- Is the system single-tenant or multi-tenant?
- If multi-tenant, what isolation guarantees are required for data, authentication, and rate limits?
- What is the practical definition of inventory accuracy and reconciliation variance?

### Success metrics realism

- API latency target p95 <= 300 ms under 1000 concurrent users applies to which endpoints?
- What dataset size is assumed (SKUs, branches, daily orders)?
- What throughput or requests per second are expected?
- What is the acceptable freshness of cached dashboard statistics and inventory snapshots?

### Product and feature clarity

- What inventory allocation policy will be used (FIFO or none)?
- Can the system automatically choose another branch if stock is unavailable?
- Can orders be split across branches?
- Are orders internal only or do they include customer information?
- Do stock adjustments require approval or only audit logging?

### Technical contract

- What authentication mechanism will be used between SPA and API (JWT, OAuth2, or session cookies)?
- Is Single Sign-On required?
- What is the idempotency strategy for API requests?

---

## 2. Missing scenarios

### Inventory and movements

- Transfer lifecycle: created -> shipped -> received
- Partial receiving
- Damaged items during transfer
- Transfer cancellation or reversal

### Units and measurement

- Unit of measure conversions (each, box, case)
- Rounding rules
- Whether system supports single unit per SKU

### Adjustments

- Adjustments linked to source documents
- Attachments or notes for audits

### Concurrency cases

- Two adjustments happening simultaneously
- Conflicting cycle counts

### Orders

- Partial fulfillment
- Partial cancellation
- Backorders
- Editing confirmed orders
- Reservation handling

### Returns

- Return to same branch or different branch
- Restockable vs non-restockable returns

---

## 3. Potential risks

### Technical risks

- Cache invalidation errors causing overselling
- Reservation expiration race conditions
- Ledger and balance mismatch

### Operational risks

- Missing monitoring tools
- Branch deactivation affecting active orders

### Performance risks

- Undefined load profile
- Excessive product search cache combinations

### Security risks

- Password reset enumeration
- Missing authorization checks

---
