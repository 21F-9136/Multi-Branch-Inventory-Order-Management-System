# Multi-Branch Inventory & Order Management System (Modular Monolith)

## 1. Problem Statement

Organizations operating across multiple branches commonly face fragmented inventory visibility, inconsistent stock deduction rules, and slow/manual reconciliation between purchasing, sales, and fulfillment. These issues produce:

- Stockouts and overselling due to stale or branch-siloed stock data
- Excess inventory and avoidable holding costs from poor demand/stock insights
- Low traceability for stock movements (who/what/when/why), making audits and shrinkage investigations difficult
- Operational bottlenecks in order processing and fulfillment when concurrency increases (e.g., campaigns, peak hours)
- Inconsistent access control and accountability across roles and branches

The core challenge is to provide real-time-ish, auditable, branch-aware inventory and order workflows that remain fast and reliable at 1000+ concurrent users.

## 2. Business Motivation

This system is a foundational operational platform for multi-branch businesses where inventory accuracy directly impacts revenue, customer experience, and cost. The business goals are to:

- Protect revenue by reducing oversells, cancellations, and delayed fulfillment
- Reduce costs by preventing overstock, enabling smarter replenishment, and shrinking manual back-office work
- Improve customer experience via reliable availability, faster processing, and predictable fulfillment
- Strengthen governance with role-based access control (RBAC), audit trails, and consistent policies across branches
- Enable scalable growth (more branches, more SKUs, more users) without re-architecting into distributed services prematurely

## 3. High-Level Solution

Build a production-grade, scalable modular monolith with clear domain boundaries and an event-driven backbone.

- Backend: CodeIgniter 4 modular architecture (domain-oriented modules with strict boundaries)
- Frontend: Vue 3 SPA with role-aware navigation and responsive dashboards
- API layer: RESTful APIs connecting the Vue SPA frontend with the CodeIgniter backend
- Data: MySQL as system of record with strong transactional guarantees for inventory reservation/deduction
- Performance: Redis for caching (hot reads and derived views)
	- Dashboard statistics (pre-aggregated KPIs for fast dashboards)
	- Product search results (cached queries for common filters/sorts)
	- Branch inventory snapshots (branch/SKU availability views to reduce read load)
- Reliability: Event-driven architecture inside the monolith (transactional outbox pattern) to publish domain events safely
- Async workloads: Queue workers for heavy/slow tasks (report aggregation, reconciliation)
- Deployment: Dockerized services (app, queue workers, MySQL, Redis) with environment-based configuration
- Quality: PHPUnit automated tests for domain logic and critical flows; JMeter load tests for concurrency and throughput targets

Key architectural principle: keep one deployable unit (monolith) while enforcing modularity and asynchronous processing to achieve scale and maintainability.

## 4. Core Capabilities

- Multi-branch inventory tracking
	- Branch-scoped stock on hand, available, reserved
	- Stock transfers between branches and receiving workflows

- Product catalog management
	- SKU/product master data, variants, and barcodes
	- Category/brand attributes and searchable catalog

- Order processing
	- Branch-aware order routing and fulfillment assignment
	- Order lifecycle (draft → confirmed → allocated/reserved → fulfilled → completed/cancelled/returned)
	- Idempotent APIs for external integrations and retries

- Inventory deduction and stock movement tracking
	- Reservation/allocation to prevent overselling under concurrency
	- Stock ledger (append-only movements) for full traceability
	- Support for adjustments, returns, damaged goods, and cycle counts

- Reporting dashboards
	- Operational dashboards (stockouts risk, aging stock, order throughput, fulfillment SLAs)
	- Finance/ops aligned summaries (movement history, shrinkage indicators, branch comparisons)
	- Scheduled reporting via queue workers

- Role-based access control (RBAC)
	- Fine-grained permissions by role and branch scope
	- Audit logging for sensitive actions (stock adjustments, cancellations, role changes)

- High performance for 1000+ concurrent users
	- Read optimization via Redis caching and precomputed views
	- Async processing for heavy workloads
	- Concurrency-safe inventory operations (transactions, locks, idempotency keys)

## 5. Expected Benefits

- Higher inventory accuracy and fewer oversells/stockouts
- Faster order throughput and more predictable fulfillment during peak load
- Lower operational cost through reduced manual reconciliation and fewer error-driven rework loops
- Better auditability and governance via stock ledger + RBAC + action trails
- Scalable growth path: add modules/queues/caches incrementally while keeping a single deployable system
