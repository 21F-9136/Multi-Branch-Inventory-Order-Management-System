## 3. Architectural Explanation

This system should be implemented as a modular monolith to keep deployment and operations simple while still enforcing clean domain boundaries for maintainability. CodeIgniter 4 remains the single backend deployable unit; domain logic is grouped into modules (Catalog, Inventory, Orders, Reporting, Auth/RBAC), and modules only interact through well-defined service interfaces and domain events.

A RESTful API layer is the contract between the Vue 3 SPA and backend. The SPA remains a thin client: it orchestrates user workflows, but all business rules (reservation, deduction, permission checks, audit logging) execute on the backend.

MySQL (InnoDB) is the system of record for all write models: orders, inventory balances, reservations, and the append-only stock movement ledger. Concurrency correctness is achieved with database transactions and row-level locking for balance rows (per branch + SKU) during reservation/deduction.

The MySQL schema should separate “current state” tables (fast reads/writes) from “history” tables (auditability):

- Current state tables
  - branches
  - products, skus (unique constraints on sku_code and barcode)
  - inventory_balances (unique (branch_id, sku_id); numeric columns for on_hand and reserved; available is derived)
  - reservations (per order_line, includes status and expiry)
  - orders, order_lines
  - users, roles, permissions, user_roles, role_permissions, user_branches
- History/audit tables
  - stock_movements (append-only ledger: branch_id, sku_id, qty_delta, movement_type, reference_type/id, actor_id, created_at, reason)
  - audit_logs (security and sensitive actions)
  - outbox_events (event_type, payload_json, status, available_at, attempts, created_at)

Minimum indexing strategy (kept pragmatic):

- inventory_balances: unique(branch_id, sku_id)
- stock_movements: (branch_id, sku_id, created_at), plus (reference_type, reference_id)
- orders: (branch_id, status, created_at)
- reservations: (branch_id, status, expires_at)
- skus: unique(sku_code), unique(barcode), plus fulltext/compound indexes for common search fields as needed

Redis is used to reduce read load and keep the SPA responsive at high concurrency. Only derived views are cached (dashboard KPIs, product search results, branch inventory snapshots). All availability decisions remain strongly consistent via MySQL transactions; the cache is never the source of truth for stock deduction.

Event-driven behavior is implemented inside the monolith using a transactional outbox. Write transactions append events to an outbox table; a worker reliably publishes them for asynchronous processing (cache invalidation, report aggregation, reconciliation checks) without risking lost events.

Docker containerization provides repeatable environments and a straightforward deployment workflow. Queue workers run as separate containers and can be scaled independently from the API runtime.

Deployment considerations (simple but production-oriented):

- Hosting limitations
  - Works on a single VM with Docker Compose for early production.
  - Can scale out behind a load balancer by adding more API containers and workers.
- Server requirements (baseline)
  - Start with 4–8 vCPU and 16 GB RAM for the API + workers on moderate datasets; size based on measured JMeter profiles.
  - MySQL benefits from dedicated resources (separate host/container and fast disk).
- Scalability design
  - Stateless API containers (no local sessions); store auth/session state in a token or Redis as needed.
  - Scale workers independently based on queue depth.
  - Keep Redis as a shared cache (single instance or managed service) and MySQL as the source of truth.
- Deployment workflow
  - Build images -> run migrations -> deploy API -> deploy workers -> monitor outbox backlog, queue depth, and DB performance.

## 4. System Patterns

### A. System Architecture

- Modular monolith: a single backend deployable with strict module boundaries.
- Layered approach within each module:
  - Controllers (REST API) -> Services (application logic) -> Domain (entities/value objects/rules) -> Repositories (persistence).
- Event-driven inside the monolith:
  - Domain events emitted by services.
  - Transactional outbox persists events for reliable async processing.

### B. Key Technical Decisions

- Backend framework: CodeIgniter 4
  - Rationale: fits the required stack; fast to build; supports a pragmatic module structure.
- API style: RESTful JSON, versioned (e.g., /api/v1)
  - Rationale: stable contract for SPA; simple to test and load test.
- Database: MySQL (InnoDB)
  - Rationale: transactional integrity and row locks for safe reservation/deduction.
- Cache: Redis
  - Use cases (explicit): dashboard statistics, product search results, branch inventory snapshots.
  - Rationale: offload read-heavy endpoints, improve p95 latencies.
- Queue: Redis-backed queue + dedicated worker containers
  - Rationale: simplest operational footprint given Redis requirement; workers scale independently.
- Event reliability: transactional outbox
  - Rationale: prevents lost events and enables safe retries.
- Testing:
  - PHPUnit for unit + feature tests (domain rules, API authorization, reservation/deduction).
  - JMeter for load profiles (read-heavy dashboards, concurrent order confirms, mixed workloads).

### C. Design Patterns in Use

1. Repository pattern
   - What: abstracts persistence behind a collection-like interface.
   - Here: each module exposes repositories (e.g., InventoryBalanceRepository, OrderRepository) so domain/services do not depend on query builder details.

2. Facade pattern
   - What: provides a simplified interface over complex subsystems.
   - Here: module-level service facades (e.g., InventoryService, OrderService) hide multi-table operations (balances + reservations + ledger + outbox events) from controllers.

3. Observer (publish/subscribe) via domain events
   - What: decouples producers and consumers of events.
   - Here: services publish domain events (StockReserved, StockDeducted, ProductUpdated) and async workers handle side effects (cache invalidation, reporting aggregates, reconciliation).

### D. Component Relationships

```
Vue 3 SPA
  -> REST API (CodeIgniter Controllers)
      -> Module Services (application rules)
          -> Domain + Repositories
              -> MySQL (transactions, locks)
          -> Outbox (within same transaction)
      -> Redis (read cache only)

Outbox worker
  -> reads MySQL outbox
  -> enqueues jobs / triggers handlers

Queue workers
  -> report aggregation
  -> reconciliation checks
  -> cache invalidation/update
```

### E. Critical Implementation Paths

1. Project foundation
   - Create repo structure (backend, frontend, infra) and docker-compose for local dev.
   - Establish environment configuration (.env.example), logging, and request correlation IDs.

2. Authentication and RBAC (must be first)
   - Implement auth endpoints and server-side authorization checks.
   - Add audit log table and middleware/filter to record sensitive actions.

3. Catalog module (enables orders)
   - CRUD for products/SKUs, barcode uniqueness, and product search endpoints.
   - Add Redis caching for product search results with bounded cache keys.

4. Inventory core (the correctness backbone)
   - Implement inventory balances + append-only stock movement ledger.
   - Implement receiving and adjustments (permission-guarded, audited).
   - Add branch inventory snapshot read endpoint + caching.

5. Order reservation and deduction
   - Draft -> confirm (reserve) -> fulfill (deduct) -> cancel (release) flows.
   - Implement transactions and row locks for (branch_id, sku_id) balance rows.
   - Implement idempotency for retryable write endpoints.

6. Events + outbox + workers
   - Write outbox events inside transactions.
   - Build worker to publish/handle events and run report aggregation and reconciliation jobs.

7. Reporting dashboards
   - Define KPI queries and aggregates.
   - Cache dashboard statistics with TTL + event-driven invalidation.

8. Performance verification
   - Create JMeter scenarios for concurrency conflicts, dashboard reads, and mixed load.
   - Tune DB indexes and query plans; validate p95 targets against the defined load profile.

9. Deployment workflow
   - Build/push images; run migrations on deploy; roll out app + workers with horizontal scaling.
   - Add basic runbooks for Redis/DB/worker outages and outbox backlog handling.
