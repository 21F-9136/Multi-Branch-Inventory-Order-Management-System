# Multi-Branch Inventory & Order Management System (PRD)

## Overview

This document defines product requirements for a production-grade Multi-Branch Inventory & Order Management System delivered as a scalable modular monolith (CodeIgniter 4 + Vue 3 SPA) with MySQL as the system of record and Redis caching. It is intended to align product, engineering, QA, and operations on scope, user experience, functional requirements, and non-functional requirements (including 1000+ concurrent user performance).

## Background and problem

Multi-branch organizations often have fragmented inventory visibility, inconsistent stock deduction rules, and limited traceability of stock movements. These gaps create stockouts and overselling, excess holding costs, and operational bottlenecks during peak load.

## Goals and success metrics

### Goals

- Provide accurate, branch-aware inventory availability and traceable stock movement history.
- Enable reliable order processing workflows that prevent overselling under concurrency.
- Deliver role-based access control across branches with auditability.
- Support operational reporting dashboards that remain responsive under high load.

## Users and roles

### Personas

- System administrator: configures tenants/branches, roles, and security policies.
- Branch manager: monitors branch performance, approves adjustments/transfers, views reports.
- Inventory clerk: receives stock, performs adjustments/cycle counts, manages stock movements.
- Sales/CSR: creates and manages customer orders, checks availability.
- Warehouse staff: picks, packs, and fulfills orders (branch-level fulfillment).
- Auditor/finance: reviews movement history, approvals, and reconciliation indicators.

### Role-based access model

- Access is scoped by role and branch.
- Sensitive actions (stock adjustments, cancellations, role changes) require explicit permissions and are audited.

## In scope

- Multi-branch inventory tracking
- Product catalog management
- Order processing
- Inventory deduction and stock movement tracking (ledger)
- Reporting dashboards

## Out of scope

## Key workflows and use cases

### Inventory availability and movement

- View availability by branch and SKU (on hand, reserved, available).
- Record receiving (increase stock) and adjustments (increase/decrease with reason).
- Transfer stock between branches with outbound/inbound traceability.
- Maintain an append-only stock movement ledger for audit.

### Order lifecycle

- Create order and allocate/reserve stock from a selected branch.
- Deduct stock on fulfillment, release reservations on cancellation/expiry.
- Handle returns that increase stock with movement traceability.

### Reporting

- View operational dashboards (stockout risk, order throughput) and movement history.
- Compare branches on key indicators.

## Product requirements

### Functional requirements

#### Authentication and RBAC

- Users authenticate to access the system.
- RBAC controls which modules and actions a user can perform, scoped by branch.
- All access-denied actions produce clear error messages and are logged.
- System administrators can create users, assign roles, and manage permissions.
- Users can manage their own credentials (e.g., password changes) according to policy.

#### Branch management

- Create, update, and deactivate branches.
- Assign users to one or more branches with explicit permissions.

#### Product catalog management

- Create and manage products and SKUs, including variants and barcodes.
- Search and filter the catalog by common attributes.
- Prevent duplicate SKUs/barcodes within a tenant.

#### Inventory tracking

- Maintain inventory quantities by branch and SKU.
- Support quantities: on hand, reserved, available.
- Ensure inventory updates are concurrency-safe.

#### Stock movement ledger

- Record all inventory-affecting events as immutable ledger entries.
- Ledger entries must include: branch, SKU, quantity delta, movement type, reference (order/transfer/adjustment), actor, timestamp, and reason where applicable.

#### Order processing

- Create and manage orders with line items.
- Allocate/reserve stock at order confirmation to prevent overselling.
- Fulfillment deducts stock and finalizes movement entries.
- Cancellation releases reserved stock and records a movement entry where required.
- API operations that can be retried must support idempotency.

#### Reporting dashboards

- Provide dashboards and reports for:
  - stockout risk
  - inventory aging indicators (where derivable from movements)
  - order throughput and fulfillment SLAs
  - branch comparisons

### Non-functional requirements

#### Performance and scalability

- Support 1000+ concurrent users with stable latency targets defined in success metrics.
- Use Redis caching for read-heavy workloads, specifically:
  - dashboard statistics: cache pre-aggregated KPIs with a short TTL and explicit invalidation via domain events
  - product search results: cache common filter/sort result sets and/or query fragments with bounded cardinality
  - branch inventory snapshots: cache branch/SKU availability views to reduce repeated reads during high traffic

#### API layer

- Provide a RESTful API layer between the Vue 3 SPA and CodeIgniter 4 backend.
- APIs must be versioned and documented.
- APIs must return consistent error formats and support pagination for list endpoints.

#### Data integrity

- MySQL is the source of truth for orders, inventory, and the stock ledger.
- Inventory reservation/deduction must be transactionally consistent.

#### Reliability and asynchronous processing

- Use an event-driven approach within the monolith for internal domain events.
- Use queue workers for heavy/slow tasks (report aggregation, reconciliation).
- Use a transactional outbox pattern (or equivalent) to ensure domain events are not lost.

#### Security

- Enforce least privilege in RBAC.
- Protect against common web threats (CSRF where applicable, input validation, rate limiting where applicable).
- Store sensitive credentials securely (hashed passwords; secrets via environment variables).

#### Observability and operations

- Provide structured logs with correlation IDs for API requests.
- Track key business events (order created, stock reserved, stock deducted) for troubleshooting.

## User stories and acceptance criteria

### Authentication and access control

#### US-001: user login

As a user, I want to log in to the system so that I can access features permitted by my role.

Acceptance criteria:
- Given valid credentials, when I log in, then I receive an authenticated session/token and can access authorized pages.
- Given invalid credentials, when I log in, then I see a generic error message and no session/token is created.
- Given multiple failed login attempts, when a threshold is exceeded, then the account is temporarily rate-limited or login is throttled.

#### US-002: user logout

As a user, I want to log out so that my session is ended on shared devices.

Acceptance criteria:
- When I log out, then my session/token is invalidated and protected pages require re-authentication.

#### US-003: enforce RBAC permissions

As a system administrator, I want actions to be restricted by role and branch so that users only perform permitted operations.

Acceptance criteria:
- Given a user without permission, when they attempt a restricted API action, then the API returns a permission error and the action does not occur.
- Given a user with permission limited to Branch A, when they attempt to act on Branch B resources, then the API denies the request.

#### US-004: audit sensitive actions

As an auditor, I want sensitive actions to be audited so that I can trace who performed them.

Acceptance criteria:
- Stock adjustments, order cancellations, and role changes create an audit record with actor, timestamp, branch scope, and before/after where applicable.

### Branch management

#### US-005: create and maintain branches

As a system administrator, I want to create, update, and deactivate branches so that the system reflects current operations.

Acceptance criteria:
- Creating a branch requires a unique branch code/name.
- Deactivating a branch prevents new orders and inventory movements for that branch while retaining historical data.

#### US-006: assign users to branches

As a system administrator, I want to assign users to one or more branches so that access is scoped correctly.

Acceptance criteria:
- A user can be assigned to one or multiple branches.
- Branch assignments affect visible data in the SPA and authorization in the API.

### Product catalog

#### US-007: create and update products and SKUs

As an inventory clerk, I want to create and update products/SKUs so that items can be stocked and sold.

Acceptance criteria:
- SKU identifiers and barcodes are unique.
- Updating product attributes does not retroactively change historical stock movement records.

#### US-008: search products

As a sales user, I want to search products so that I can quickly add items to an order.

Acceptance criteria:
- Search returns results filtered by query and supports pagination.
- Common searches are performant under load (supported via Redis caching).

#### US-009: handle duplicate identifiers

As a system, I want to prevent duplicate SKUs/barcodes so that catalog data remains consistent.

Acceptance criteria:
- Creating or updating a SKU/barcode to a value that already exists is rejected with a clear validation error.

### Inventory and stock movements

#### US-010: view inventory by branch

As a branch manager, I want to view inventory on hand/reserved/available by branch so that I can make fulfillment decisions.

Acceptance criteria:
- Inventory views display on hand, reserved, and available quantities per SKU.
- Inventory views for high-traffic screens use cached branch inventory snapshots where appropriate.

#### US-011: record receiving

As an inventory clerk, I want to record receiving so that stock on hand increases and is auditable.

Acceptance criteria:
- Receiving creates one or more stock movement ledger entries.
- Receiving updates inventory quantities transactionally.

#### US-012: perform stock adjustment with reason

As an inventory clerk, I want to adjust stock with a required reason so that shrinkage and corrections are traceable.

Acceptance criteria:
- Adjustments require a reason code/text.
- Adjustments create ledger entries and an audit record.
- Users without permission cannot adjust stock.

#### US-013: transfer stock between branches

As an inventory clerk, I want to transfer stock between branches so that inventory can be rebalanced.

Acceptance criteria:
- A transfer produces an outbound movement from the source branch and an inbound movement to the destination branch.
- Transfer references link outbound/inbound records for traceability.
- Transfers cannot result in negative on hand at the source branch.

#### US-014: prevent negative available stock

As a system, I want to prevent operations that would make available stock negative so that overselling is prevented.

Acceptance criteria:
- If an action would reduce available stock below zero, the transaction is rejected.
- The system returns a clear error describing insufficient stock.

#### US-015: view stock movement ledger

As an auditor, I want to view a stock movement ledger so that I can trace all inventory changes.

Acceptance criteria:
- Ledger entries are immutable and can be filtered by branch, SKU, date range, and movement type.
- Ledger entries show actor and reference identifiers.

### Order processing

#### US-016: create an order

As a sales user, I want to create an order with line items so that customers can purchase products.

Acceptance criteria:
- An order can be created in draft state.
- Line items validate SKU existence and quantity > 0.

#### US-017: confirm an order and reserve stock

As a sales user, I want to confirm an order so that stock is reserved and not oversold.

Acceptance criteria:
- Confirming an order reserves stock for the chosen branch.
- Reservation updates reserved/available quantities transactionally.
- If stock is insufficient, confirmation fails and no partial reservation is left behind.

#### US-018: handle concurrent confirmations safely

As a system, I want concurrent order confirmations to be safe so that two users cannot reserve the same stock.

Acceptance criteria:
- Under concurrent confirmations for the same SKU/branch, reservations are serialized/transactionally safe.
- At most one confirmation succeeds when remaining available stock is insufficient for both.

#### US-019: fulfill an order and deduct stock

As warehouse staff, I want to fulfill an order so that stock is deducted and the order progresses.

Acceptance criteria:
- Fulfillment deducts on hand stock and reduces reserved accordingly.
- Fulfillment creates stock movement ledger entries referencing the order.

#### US-020: cancel an order and release reservations

As a sales user, I want to cancel an order so that reserved stock is released and inventory becomes available again.

Acceptance criteria:
- Cancelling a confirmed order releases reserved stock transactionally.
- Cancellation is audited.

#### US-021: process returns

As an inventory clerk, I want to process returns so that stock increases with traceability.

Acceptance criteria:
- Returns create stock movement entries with reference to the original order.
- Return quantity cannot exceed fulfilled quantity for the order line.

#### US-022: support idempotent order APIs

As an integrator, I want to safely retry order requests so that network failures do not create duplicates.

Acceptance criteria:
- For idempotent endpoints, repeated requests with the same idempotency key produce the same outcome without duplicate orders/reservations.

### Reporting and dashboards

#### US-023: view operational dashboards

As a branch manager, I want dashboards for stockout risk and order throughput so that I can act quickly.

Acceptance criteria:
- Dashboards load within the defined performance targets under load.
- Dashboard statistics are served from Redis cache where appropriate, with freshness controls.

#### US-024: view branch comparisons

As an operations manager, I want to compare branches so that I can identify performance differences.

Acceptance criteria:
- Reports allow selecting a date range and branches.
- Results are paginated/exportable as needed in later phases.

#### US-025: schedule reporting aggregation

As a system, I want heavy report aggregation to run asynchronously so that user-facing requests stay fast.

Acceptance criteria:
- Report aggregation runs via queue workers.
- Failures are retried and logged.

### Error handling and edge cases

#### US-026: handle permission denied gracefully

As a user, I want clear messaging when I lack permissions so that I understand why an action failed.

Acceptance criteria:
- UI shows a non-sensitive error message.
- API returns a consistent permission error schema.

#### US-027: prevent actions on deactivated branches

As a system, I want to prevent new operational actions on deactivated branches so that data stays consistent.

Acceptance criteria:
- New orders and stock movements are rejected for deactivated branches.
- Read-only access to historical data remains available to permitted users.

### User and role administration

#### US-028: reset password

As a user, I want to reset my password so that I can regain access if I forget it.

Acceptance criteria:
- When I request a password reset, then the system provides a secure reset mechanism that does not reveal whether the account exists.
- When I complete a valid reset, then my old credentials no longer work and I can log in with the new password.

#### US-029: change password

As a user, I want to change my password so that I can maintain account security.

Acceptance criteria:
- Changing a password requires confirming the current password.
- The new password must meet the configured complexity policy.

#### US-030: create and deactivate users

As a system administrator, I want to create and deactivate users so that access matches staffing changes.

Acceptance criteria:
- Creating a user requires a unique identifier (e.g., email/username).
- Deactivating a user prevents future logins while retaining historical audit attribution.

#### US-031: manage roles and permissions

As a system administrator, I want to manage roles and permissions so that access aligns to responsibilities.

Acceptance criteria:
- Updating role permissions takes effect immediately for subsequent API requests.
- Role changes are audited with actor, timestamp, and before/after permission sets.

### Order changes and reservation behavior

#### US-032: edit a draft order

As a sales user, I want to edit a draft order so that I can correct items and quantities before confirming.

Acceptance criteria:
- Draft orders can be updated (add/remove items, change quantities) without affecting inventory.
- Confirmed orders cannot be edited in ways that would invalidate existing reservations without an explicit re-confirmation flow.

#### US-033: expire stale reservations

As a system, I want reservations to expire after a configurable time so that stock is not held indefinitely.

Acceptance criteria:
- Reservations older than the configured TTL are released automatically.
- Reservation expiry is recorded for audit/troubleshooting and does not leave inventory in an inconsistent state.

### Inventory counting and reconciliation

#### US-034: record cycle count

As an inventory clerk, I want to record a cycle count so that discrepancies can be corrected with traceability.

Acceptance criteria:
- A cycle count captures the counted quantity, timestamp, actor, and reference notes.
- Discrepancies result in an adjustment entry with an explicit linkage to the cycle count.

### Audit and compliance

#### US-035: view audit logs

As an auditor, I want to view audit logs so that I can review sensitive actions and approvals.

Acceptance criteria:
- Audit logs can be filtered by date range, actor, branch, and action type.
- Audit log entries are immutable and display sufficient context to reconstruct the action.

## API requirements

### API principles

- RESTful resources for branches, products/SKUs, inventory views, movements, orders, and reports.
- Versioned base path (e.g., /api/v1).
- Consistent response envelopes for lists and errors.

### Security and auth

- Authenticated access required for all non-public endpoints.
- Authorization enforced server-side for every request.

## Data requirements

### Core entities

- Branch
- User, role, permission, and user-branch assignment
- Product and SKU
- Inventory balance (per branch/SKU)
- Reservation (per order line where applicable)
- Stock movement ledger (append-only)
- Order and order line items

### Data quality rules

- SKU/barcode uniqueness
- No negative on hand or available quantities
- Ledger entries immutable

## Dependencies and constraints

- Backend must be CodeIgniter 4.
- Frontend must be Vue 3 SPA.
- MySQL is system of record; Redis is for caching.
- Deployment uses Docker containers.
- Automated tests use PHPUnit; load tests use JMeter.

## Risks and mitigations

- Risk: overselling under concurrency.
  - Mitigation: transactional reservations and safe deduction semantics; idempotent APIs.
- Risk: cache serving stale availability.
  - Mitigation: cache only derived views; keep MySQL as source of truth; short TTL and event-driven invalidation for snapshots.
- Risk: reporting queries degrade performance.
  - Mitigation: async report aggregation via queue workers; cached dashboard KPIs.

## Testing and validation

- Unit and integration tests (PHPUnit) for:
  - reservation and deduction logic
  - movement ledger correctness and immutability
  - RBAC authorization checks
- Load testing (JMeter) scenarios for:
  - concurrent order confirmations on the same SKU/branch
  - dashboard reads (cached) and inventory reads
  - mixed read/write workload at 1000+ concurrent users
